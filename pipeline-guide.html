The Pipeline

The evaluation runs in three stages. Each stage is independent—you can re-run any stage without affecting the others.

Stage 1: Embed

We generate embeddings for every document and query in each dataset. This is the expensive part—API calls to 16 different embedding providers.

Each model gets two files per dataset: corpus_{model}.npy and queries_{model}.npy. We store these as NumPy arrays because they're large (millions of floats) and we need fast cosine similarity later.

L2 Normalization: All embeddings are L2-normalized before saving. This lets us use dot product instead of cosine similarity, which is faster.

Stage 2: Evaluate

For each query, we retrieve the top-k documents using cosine similarity and compute standard IR metrics:

  • NDCG@10 — Normalized Discounted Cumulative Gain. Measures ranking quality.
  • Recall@10 — What fraction of relevant documents appear in top 10.

These metrics require ground truth labels (qrels). Not every dataset has them, so some datasets only get LLM judge evaluation.

Stage 3: LLM Judge

This is where it gets interesting. We use GPT-5 to compare search results from two models head-to-head.

For each query, we show the judge the top-5 results from Model A and Model B. The judge picks which ranking is more relevant: A, B, or TIE.

Given a query and two search result lists, which is more relevant? Query: "What is the best way to invest in index funds?" Ranking A: 1. Index funds are a type of mutual fund that tracks... 2. The best index funds for beginners include... 3. Vanguard's Total Stock Market Index Fund... Ranking B: 1. Investing in stocks requires careful analysis... 2. Index funds provide broad market exposure... 3. The S&P 500 has historically returned... Answer: A, B, or TIE.

Position Bias: LLMs tend to favor the first option they see. We randomly swap A/B positions and track which was which, then map the judgment back to the actual models.

Elo Calculation

We convert win/loss/tie records into Elo ratings using the standard formula:

  • Every model starts at 1500
  • K-factor of 32 (how much ratings change per game)
  • Ties count as 0.5 wins for each side
  • We run 10 iterative passes to stabilize ratings

The nice thing about Elo is that it's transitive. If Model A beats B, and B beats C, then A should beat C—and the ratings reflect that even if A and C never played directly.

Adding a New Model

Want to add your model to the leaderboard? Four steps.

Step 1: Add to model-info.json

This file contains metadata about each model. Add an entry with the model's name, provider, dimensions, and any other relevant info.

// results/model-info.json { "name": "your-model-name", "display_name": "Your Model Display Name", "provider": "YourCompany", "license": "Proprietary", "cost_per_1m_tokens": 0.10, "dimension": 1024, "about_model": "Brief description of the model." }

Step 2: Add embedding client

If your model uses a new API (not OpenAI, Voyage, Cohere, etc.), add a client class to pipeline/stages/embed.py.

class YourModelClient: def __init__(self, api_key: str, model_name: str): self.api_key = api_key self.model_name = model_name self.api_url = "https://api.yourcompany.com/embed" def embed_corpus(self, texts: List[str], batch_size: int = 64) -> np.ndarray: embeddings = [] for i in range(0, len(texts), batch_size): batch = texts[i:i + batch_size] response = requests.post( self.api_url, headers={"Authorization": f"Bearer {self.api_key}"}, json={"input": batch, "model": self.model_name} ) data = response.json() embeddings.extend(data["embeddings"]) return np.array(embeddings) def embed_queries(self, texts: List[str], batch_size: int = 64) -> np.ndarray: # Same as embed_corpus, or use input_type="query" if supported return self.embed_corpus(texts, batch_size)

Step 3: Add to config.yaml

Register the model in the config file so the pipeline knows how to instantiate it.

# config.yaml embedders: - name: "your-model-name" provider: "yourcompany" model: "your-model-v1" api_key_env: "YOUR_API_KEY"

Step 4: Run the pipeline

Set your API key and run the add_model script. It handles everything: generating embeddings, running evaluations, computing Elo.

# Set your API key export YOUR_API_KEY="sk-..." # Run the pipeline for your model python pipeline/add_model.py --model your-model-name

The script will:

  1. Generate embeddings for all 7 datasets
  2. Calculate NDCG and Recall metrics
  3. Run LLM judge comparisons against all existing models
  4. Recompute Elo rankings
  5. Update benchmarks.json

Configuration Reference

Setting Value Why
Queries per comparison 10 Balance between cost and statistical significance
Top-k for judge 5 Enough to see ranking quality without overwhelming the judge
Document truncation 200 chars Keep judge prompts under token limits
Judge max_tokens 10 Force short responses (A, B, or TIE)
Elo K-factor 32 Standard chess rating sensitivity
Elo initial 1500 Standard starting point
Elo iterations 10 Enough passes for ratings to converge

ZEMBED_API_KEY=ze_... # LLM Judge (Azure OpenAI) AZURE_API_KEY=... AZURE_RESOURCE_NAME=https://your-resource.openai.azure.com/openai/v1/ AZURE_DEPLOYMENT_ID=gpt-5

Cost estimate: Adding a new model costs roughly $5-15 in embedding API calls (depending on the provider) plus $10-20 in LLM judge calls for ~1000 comparisons.

Reusing Judgments

LLM judge results are cached in results/llm_judge/. Each comparison gets its own file:

fiqa_voyage-4_vs_text-embedding-3-large.json msmarco_zembed-1_vs_voyage-4.json ...

If a judgment file exists, the pipeline skips that comparison. This means:

  • Re-running is cheap—only new comparisons are computed
  • Results are reproducible—same judgments, same Elo
  • You can manually inspect or edit individual comparisons

Never regenerate existing judgments. The whole point of caching is consistency. If you re-run a comparison, you might get different results due to LLM randomness, which would make historical comparisons invalid.

The Bottom Line

The pipeline is designed to be modular and extensible. Add a client, update the config, run the script. The hard part is already done—the evaluation framework, the judge prompts, the Elo calculation.

If you add a model and get interesting results, let us know. We're always looking for new models to benchmark.

# Quick start git clone https://github.com/agentset/embedding-leaderboard cd embedding-leaderboard cp .env.example .env # Edit .env with your API keys python pipeline/add_model.py --model your-model
